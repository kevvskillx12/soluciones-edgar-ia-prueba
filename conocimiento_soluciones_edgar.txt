Pregunta frecuente: ¿Por qué está justificado el uso de IA en Soluciones Edgar?

El uso de IA en Soluciones Edgar está justificado porque no se utiliza solamente como un buscador de información. La IA se usa como apoyo administrativo para analizar información de pedidos reales, redactar reportes de cierre, resumir errores, explicar estados de trámites y generar conclusiones útiles para el administrador.

El sistema no depende de la IA para hacer operaciones críticas como crear pedidos, descontar saldo, consultar la base de datos, cambiar estados o procesar servicios por API externa. Esas tareas las realiza Laravel mediante lógica normal del sistema.

La IA se usa donde aporta más valor: convertir datos técnicos y operativos en explicaciones claras. Por ejemplo, cuando el administrador pide un reporte de los últimos trámites, Laravel consulta la base de datos real, calcula los totales, identifica pedidos completados, fallidos, pendientes o procesados por API externa, y después Ollama redacta un resumen administrativo.

Esto evita que el administrador tenga que revisar pedido por pedido y escribir manualmente un cierre del día. También ayuda a detectar errores, revisar trámites pendientes y explicar el funcionamiento interno del sistema.

El uso de IA local con Ollama también está justificado porque permite trabajar sin depender de servicios externos de IA, protege mejor la información del sistema y reduce costos por consumo de tokens en la nube.

En resumen, la IA en Soluciones Edgar está justificada porque funciona como asistente administrativo local. No reemplaza la lógica del sistema, sino que ayuda a resumir, explicar y documentar información real de los trámites.

## 1. RESUMEN GENERAL DEL SISTEMA

Soluciones Edgar es una plataforma web para gestionar trámites y servicios digitales. El sistema permite que los usuarios se registren, consulten servicios, soliciten trámites, paguen con saldo interno, revisen el estado de sus pedidos y descarguen resultados cuando el trámite está completado.

El sistema tiene un panel administrativo construido con Filament. Desde ese panel el administrador puede gestionar usuarios, servicios, pedidos, solicitudes de depósito, transacciones, reportes y el chat administrativo con IA local.

El sistema maneja usuarios, administradores, servicios, pedidos, saldo, depósitos, transacciones, categorías, soporte, actividad del sistema, automatización por API externa, reportes de cierre e integración de IA local.

---

## 2. TECNOLOGÍAS DEL SISTEMA

Soluciones Edgar usa Laravel y PHP como base del backend.

El proyecto usa Eloquent ORM para manejar modelos y relaciones de base de datos.

Usa migrations de Laravel para crear y modificar tablas.

Usa seeders para cargar datos iniciales como el usuario administrador y el catálogo de servicios.

Usa Filament v3 para construir el panel administrativo y los recursos de gestión.

Usa Livewire Volt para páginas de autenticación como login, registro, recuperación de contraseña, confirmación de contraseña y verificación de correo.

Usa el sistema de notificaciones de Laravel para enviar correos de verificación y recuperación de contraseña.

Usa almacenamiento local público y almacenamiento compatible con S3 para guardar y descargar archivos de resultado.

Usa Python para el sistema RAG.

Usa ChromaDB como base de datos vectorial.

Usa el modelo de embeddings all-MiniLM-L6-v2 para convertir fragmentos de texto en vectores.

Usa Ollama de forma local con el modelo llama3.2:1b para generar respuestas.

Usa la ruta /ia-test para conectar el chat del sistema con comandos administrativos, RAG y Ollama.

---

Pregunta frecuente: ¿Qué tecnologías usa el sistema?

El sistema usa Laravel, PHP, Eloquent ORM, migrations, seeders, Filament v3, Livewire Volt, notificaciones de Laravel, almacenamiento local o S3, Python, ChromaDB, embeddings all-MiniLM-L6-v2, Ollama local y el modelo llama3.2:1b.

## 3. USUARIOS Y ADMINISTRADORES

La tabla users guarda la información principal de los usuarios.

Cada usuario tiene nombre, correo, contraseña, saldo y datos adicionales como empresa, teléfono, tipo de cuenta y verificación de empresa.

El campo is_admin indica si un usuario es administrador.

El campo balance guarda el saldo disponible del usuario.

El acceso al panel administrativo se controla con el método canAccessPanel del modelo User. Si el panel es admin, solo pueden entrar usuarios con is_admin en true.

Un usuario puede tener muchos pedidos, muchas solicitudes de depósito y muchas transacciones.

---

Pregunta frecuente: ¿Cómo se identifica un administrador?

Un administrador se identifica con el campo is_admin en true dentro de la tabla users. El método canAccessPanel del modelo User revisa ese campo para permitir o negar acceso al panel admin de Filament.

---

## 4. SALDO, DEPÓSITOS Y TRANSACCIONES

El saldo del usuario se guarda en el campo balance de la tabla users.

El saldo inicial de un usuario es 0.

El sistema puede agregar saldo con los métodos credit o addBalance del modelo User.

El sistema puede descontar saldo con el método subtractBalance del modelo User.

Cuando se agrega saldo, se registra una transacción de tipo deposit o refund, dependiendo del origen.

Cuando se descuenta saldo por una compra, se registra una transacción de tipo purchase.

Si el usuario no es administrador y no tiene saldo suficiente, el sistema detiene la operación y muestra o lanza el error Saldo insuficiente.

Las transacciones se guardan en la tabla transactions.

La tabla transactions tiene user_id, type, amount, reference_type, reference_id, description y timestamps.

Los tipos de transacción son deposit, purchase y refund.

---

Pregunta frecuente: ¿Cómo funciona el saldo del usuario?

El saldo del usuario se guarda en el campo balance de la tabla users. El sistema agrega saldo con credit o addBalance, y descuenta saldo con subtractBalance. Cuando se descuenta saldo se registra una transacción de tipo purchase. Si el usuario no es administrador y no tiene saldo suficiente, el sistema detiene la operación con el error Saldo insuficiente.

## 5. SOLICITUDES DE DEPÓSITO

Las solicitudes de depósito se guardan en la tabla deposit_requests.

Cada solicitud de depósito pertenece a un usuario.

Una solicitud de depósito guarda user_id, bank_name, tracking_key, amount, proof_image_path, status, admin_notes y timestamps.

tracking_key representa la clave de rastreo del depósito.

proof_image_path guarda la ruta del comprobante de pago.

Los estados de una solicitud de depósito son pending, approved y rejected.

pending significa que la solicitud está pendiente de revisión.

approved significa que el depósito fue aprobado y el saldo puede agregarse al usuario.

rejected significa que el depósito fue rechazado.

---

Pregunta frecuente: ¿Qué hace DepositRequestResource?

DepositRequestResource es el recurso de Filament que administra las solicitudes de depósito. Permite listar, crear, editar y eliminar solicitudes. Cada solicitud guarda usuario, banco, clave de rastreo, monto, comprobante, estado y notas administrativas. Los estados de depósito son pending, approved y rejected.

## 6. SERVICIOS

Los servicios se guardan en la tabla services.

Cada servicio puede tener código, nombre, descripción, precio, costo, tipo de servicio, categoría, imagen, horario activo, tiempo de procesamiento, estado activo y formulario dinámico.

El campo price indica el precio que paga el usuario.

El campo cost indica el costo interno del servicio.

El campo is_active indica si el servicio está activo.

El campo category_id relaciona el servicio con una categoría.

El campo form_schema guarda la estructura JSON del formulario dinámico que debe llenar el usuario.

Un servicio puede tener muchos pedidos.

Un servicio pertenece a una categoría.

---

Pregunta frecuente: ¿Qué servicios ofrece Soluciones Edgar?

Soluciones Edgar ofrece servicios de actas, SAT, IMSS, servicios generales, Infonavit y trámites vehiculares. Entre los servicios registrados están acta de nacimiento, acta de defunción, acta de divorcio, acta de matrimonio, CSF con CURP, CSF con RFC e IDCIF, localizar IDCIF, constancia de vigencia de derechos NSS, localizar NSS, semanas cotizadas, localizar AFORE, CURP actualizada, recibo CFE, antecedentes no penales, servicios de Infonavit, formatos de pago de tenencia CDMX y EDOMEX, y hoja REPUVE.

Pregunta frecuente: ¿Qué hace ServiceResource?

ServiceResource es el recurso de Filament encargado de administrar el catálogo de servicios. Permite crear, listar, editar y eliminar servicios. También permite organizar servicios por categorías, manejar precios, costos, formularios dinámicos, imagen, estado activo y configuración de automatización por API externa.

## 8. PEDIDOS O TRÁMITES

Los pedidos se guardan en la tabla orders.

Cada pedido pertenece a un usuario y a un servicio.

Un pedido guarda user_id, service_id, input_data, status, result_file_path, admin_notes, price_at_purchase, service_cost_snapshot, service_price_snapshot y datos de API externa.

El campo input_data guarda la información capturada del usuario, como CURP, RFC, NSS, placa u otros datos según el servicio.

El campo result_file_path guarda la ruta del archivo de resultado.

El campo admin_notes guarda notas del administrador.

El campo price_at_purchase guarda el precio del servicio al momento de la compra.

Los campos service_cost_snapshot y service_price_snapshot guardan el costo y precio del servicio al momento de crear el pedido.

Los estados de un pedido son pending, processing, completed y rejected.

pending significa pendiente.

processing significa en proceso.

completed significa completado.

rejected significa rechazado.

---

Pregunta frecuente: ¿Cuáles son los estados de un pedido?

Los estados de un pedido son pending, processing, completed y rejected. Pending significa pendiente, processing significa en proceso, completed significa completado y rejected significa rechazado.

---

Pregunta frecuente: ¿Cómo se procesan los pedidos?

En Soluciones Edgar, los pedidos se procesan cuando un usuario solicita un servicio. Primero se crea el pedido relacionado con el usuario y el servicio. También se guardan los datos capturados en input_data, como CURP, RFC, NSS, placa u otros datos según el trámite.

Después el pedido puede manejarse de forma manual o automática.

Si el servicio es manual, el administrador revisa el pedido, realiza el trámite y sube el resultado.

Si el servicio es automático, el sistema despacha ProcessExternalOrderJob. Este Job usa ExternalOrderAutomationService para procesar el pedido por API externa.

Cuando el proveedor configurado es dhru, se usa DhruApiService. En modo simulación, DhruApiService genera una orden externa con prefijo SIM, crea un archivo de resultado, actualiza api_status a completed, marca processed_by_api como verdadero y guarda el reporte en api_report.

Si ocurre un error, el sistema guarda api_status como failed y registra el error en api_error_message. Si el pedido necesita atención humana, puede quedar como manual_review.

Cuando el proceso termina correctamente, el resultado se guarda en result_file_path y el usuario o administrador puede descargarlo.

---

## 9. AUTOMATIZACIÓN POR API EXTERNA

El sistema tiene una automatización por API externa para procesar algunos pedidos sin intervención manual.

La automatización no reemplaza todos los trámites. Solo aplica a servicios configurados como automáticos.

Los servicios pueden tener automation_type.

Los valores de automation_type son manual, semi_automatic y automatic.

manual significa que el trámite lo atiende el administrador.

semi_automatic significa que puede requerir apoyo o revisión del administrador.

automatic significa que el sistema puede enviar el pedido al flujo de API externa.

Los servicios también pueden tener external_provider, external_service_id y api_handler.

external_provider indica el proveedor externo, por ejemplo dhru.

external_service_id indica el ID del servicio en el proveedor externo.

api_handler indica qué manejador o clase se usa para procesar el servicio.

---

Pregunta frecuente: ¿Qué significa automatización por API externa?

Significa que algunos servicios pueden procesarse automáticamente usando un proveedor externo. Si el servicio está configurado como automatic, el sistema envía el pedido al flujo de API externa, guarda el estado, genera un resultado y marca el pedido como procesado por API externa.

---

## 10. DHRU API SERVICE

DhruApiService es el servicio encargado de representar la conexión con el proveedor externo Dhru.

DhruApiService tiene métodos como isEnabled, getServices, placeOrder, checkOrderStatus y getOrderResult.

Cuando DHRU_API_ENABLED está en false, DhruApiService funciona en modo simulación.

En modo simulación, el sistema no hace llamadas reales a internet.

En modo simulación, DhruApiService genera IDs externos simulados con prefijo SIM.

En modo simulación, se genera un archivo de resultado de prueba.

En modo simulación, el pedido puede quedar como completed y processed_by_api en true.

Cuando la API real esté habilitada, el servicio debe usar las credenciales configuradas en el archivo .env y consumir el proveedor real.

---

Pregunta frecuente: ¿Qué hace DhruApiService?

DhruApiService es el servicio que conecta o simula la conexión con el proveedor externo Dhru. Puede consultar servicios, colocar órdenes, revisar estados y obtener resultados. Si DHRU_API_ENABLED está en false, trabaja en modo simulación y genera resultados de prueba sin consumir internet.

---

## 11. EXTERNAL ORDER AUTOMATION SERVICE

ExternalOrderAutomationService es el orquestador de la automatización externa.

Este servicio decide qué hacer según el tipo de automatización del servicio.

Si el servicio es manual, el pedido queda para revisión manual.

Si el servicio es semi_automatic, el pedido puede quedar para revisión manual con nota.

Si el servicio es automatic, el sistema intenta procesarlo con el proveedor externo configurado.

ExternalOrderAutomationService puede usar DhruApiService cuando el proveedor es dhru.

Si el procesamiento termina correctamente, guarda información de API en el pedido.

Si falla, guarda el error en api_error_message y puede dejar el estado API como failed o manual_review.

---

Pregunta frecuente: ¿Qué hace ExternalOrderAutomationService?

ExternalOrderAutomationService decide y ejecuta el flujo de automatización de un pedido. Revisa si el servicio es manual, semiautomático o automático. Si es automático y el proveedor es dhru, llama a DhruApiService. Luego actualiza el estado API, el reporte, el archivo de resultado o el error del pedido.

---

## 12. PROCESS EXTERNAL ORDER JOB

ProcessExternalOrderJob es un Job de Laravel que procesa pedidos automáticos en segundo plano o de forma inmediata según la configuración de colas.

El Job recibe un pedido.

El Job llama a ExternalOrderAutomationService.

El Job maneja errores con try/catch.

El Job tiene reintentos y timeout.

Si QUEUE_CONNECTION está en sync, el Job se ejecuta inmediatamente al crear el pedido.

Si QUEUE_CONNECTION usa database o redis, se necesita ejecutar php artisan queue:work.

---

Pregunta frecuente: ¿Qué hace ProcessExternalOrderJob?

ProcessExternalOrderJob recibe un pedido y llama a ExternalOrderAutomationService para procesarlo por API externa. Si la cola está en sync, se ejecuta inmediatamente. Si hay un error, guarda la información para que el administrador pueda revisarla.

---

## 13. CAMPOS DE API EN PEDIDOS

Los pedidos pueden guardar campos relacionados con API externa.

processed_by_api indica si el pedido fue procesado por API externa.

api_status indica el estado del procesamiento externo.

external_provider indica el proveedor usado, por ejemplo dhru.

external_order_id indica el ID de orden generado por el proveedor externo.

api_processed_at indica la fecha y hora del procesamiento.

api_report guarda el reporte del proceso externo.

api_error_message guarda el error si algo falla.

Los estados API pueden ser pending, queued, sent, processing, completed, failed y manual_review, según la implementación del flujo.

completed significa que la API terminó correctamente.

failed significa que hubo error.

manual_review significa que necesita revisión humana.

queued significa que el pedido fue puesto en cola.

---

Pregunta frecuente: ¿Qué información guarda un pedido procesado por API?

Guarda processed_by_api, api_status, external_provider, external_order_id, api_processed_at, api_report y api_error_message. Estos campos permiten saber si el pedido fue automatizado, qué proveedor se usó, cuál fue la orden externa, qué ocurrió y si hubo errores.

## 14. DESCARGA DE RESULTADOS

El resultado de un pedido se guarda en result_file_path.

El usuario autenticado puede descargar el resultado desde la ruta /app/orders/{order}/download.

El sistema valida que el usuario sea dueño del pedido o que sea administrador.

Si el usuario no tiene permiso, el sistema devuelve error 403.

Si el pedido no tiene archivo de resultado, el sistema devuelve error 404.

La descarga puede buscar el archivo en el disco public o en el disco s3, dependiendo de dónde se haya guardado.

En modo simulación de API, el resultado puede guardarse como archivo de texto en storage/app/public/results.

---

Pregunta frecuente: ¿Cómo se descarga el resultado de un pedido?

El resultado se descarga desde /app/orders/{order}/download. El sistema valida permisos, revisa que exista result_file_path y descarga el archivo desde public o s3. Si el archivo no existe, devuelve 404.

OrderResource puede incluir acciones para procesar por API externa y acciones masivas para generar reportes de cierre.

Pregunta frecuente: ¿Qué hace OrderResource?

OrderResource administra los pedidos o trámites desde Filament. ## 20. IA LOCAL, RAG Y CHAT ADMINISTRATIVO

El sistema tiene integración con IA local mediante Ollama.

El modelo usado es llama3.2:1b.

El sistema RAG usa Python y ChromaDB.

El archivo conocimiento_soluciones_edgar.txt se ingesta con ingestar.py.

ingestar.py lee el documento, limpia el texto, lo divide en fragmentos, genera embeddings y guarda los fragmentos en ChromaDB.

rag_bridge.py recibe una pregunta, busca contexto en ChromaDB y manda el contexto a Ollama para generar una respuesta.

La ruta /ia-test conecta el chat del panel con comandos administrativos, RAG y Ollama.

El chat administrativo puede responder preguntas sobre la base de conocimiento y también ejecutar comandos especiales si el usuario es administrador.

---

Pregunta frecuente: ¿Cómo funciona la IA del sistema?

La IA funciona localmente con Ollama y el modelo llama3.2:1b. Para responder sobre el proyecto, el sistema usa RAG: primero busca fragmentos en ChromaDB y después manda ese contexto a Ollama para generar una respuesta basada en la información del sistema.

---

Pregunta frecuente: ¿Qué es RAG en este proyecto?

RAG significa Retrieval-Augmented Generation. En Soluciones Edgar se usa para que la IA no responda solo con conocimiento general, sino con información real del archivo conocimiento_soluciones_edgar.txt guardada en ChromaDB.

## 21. COMANDOS DEL CHAT ADMINISTRATIVO

El chat administrativo puede funcionar como asistente del administrador.

Algunos comandos consultan la base de conocimiento RAG.

Otros comandos consultan datos reales de la base de datos mediante AdminChatCommandService.

El comando para generar reporte de cierre no depende del TXT del RAG. Ese comando lo detecta AdminChatCommandService y consulta directamente los pedidos reales de la base de datos.

Comandos útiles:

- Que servicios ofrece Soluciones Edgar
- Cuales son los estados de un pedido
- Como funciona el saldo del usuario
- Que tecnologias usa el sistema
- Que hace DepositRequestResource
- Que hace ServiceResource
- Que hace OrderResource
- Como se procesan los pedidos
- Genera un reporte de los últimos 5 trámites
- Genera un reporte del último trámite
- Reporte de cierre
- Resume los pedidos fallidos
- Qué quedó pendiente de revisión
- Trámites en revisión manual
- Resumen de trámites completados hoy
- Cierre de hoy

---

Pregunta frecuente: ¿Por qué el comando de reporte no depende del RAG?

Porque un reporte de cierre debe usar datos reales y actuales de la base de datos, no solo texto del documento de conocimiento. Por eso AdminChatCommandService intercepta el mensaje, consulta los pedidos reales, genera el reporte y puede usar Ollama para redactar un resumen administrativo.

## 22. REPORTE DE CIERRE DE PEDIDOS

El sistema puede generar reportes de cierre para pedidos procesados.

El reporte puede crearse desde una acción masiva en OrderResource o desde el chat administrativo mediante comandos.

El reporte se guarda como archivo .txt en storage/app/public/reports.

El nombre del archivo puede tener el formato reporte-cierre-pedidos-YYYYMMDD-HHMMSS.txt.

El reporte incluye fecha, total de pedidos, completados, pendientes, en proceso, rechazados, procesados por API externa, fallidos por API, revisión manual y proveedores usados.

El reporte también incluye detalle de pedidos: pedido, servicio, usuario, estado, processed_by_api, api_status, proveedor, orden externa, reporte API y error.

El resumen final puede generarse con Ollama local.

Si Ollama no está disponible, el sistema puede generar un resumen básico con PHP.

---

Pregunta frecuente: ¿Qué contiene el reporte de cierre?

El reporte de cierre contiene resumen general de pedidos, cantidades por estado, pedidos procesados por API externa, fallos, revisión manual, proveedores utilizados, detalle de cada pedido, reportes API, errores y un resumen final generado con Ollama o con PHP si Ollama no está disponible.

## 23. JUSTIFICACIÓN DEL USO DE IA

La IA no se usa solo para buscar información que ya aparece en pantalla.

La IA se usa como asistente administrativo para explicar el sistema, recuperar información técnica del proyecto, analizar pedidos, redactar reportes de cierre y generar resúmenes administrativos.

La automatización por API externa se encarga del procesamiento automático de trámites cuando el servicio lo permite.

Ollama ayuda a redactar reportes y conclusiones sin depender de servicios externos de IA.

El RAG ayuda a que las respuestas se basen en información documentada del proyecto.

La combinación de API externa, reportes e IA local permite reducir trabajo manual, mejorar trazabilidad y dejar evidencia del proceso.

## 24. FLUJO COMPLETO DEL SISTEMA

1. El usuario inicia sesión.
2. El usuario consulta los servicios disponibles.
3. El usuario solicita un trámite.
4. El sistema guarda el pedido con usuario, servicio e input_data.
5. El sistema valida saldo.
6. Si hay saldo suficiente, descuenta el saldo y registra una transacción purchase.
7. Si el servicio es manual, queda para revisión del administrador.
8. Si el servicio es automático, se despacha ProcessExternalOrderJob.
9. El Job usa ExternalOrderAutomationService.
10. Si el proveedor es dhru, se usa DhruApiService.
11. En simulación se genera una orden SIM y un archivo de resultado.
12. El pedido se marca como completed y processed_by_api si terminó correctamente.
13. El usuario o administrador descarga el resultado.
14. El administrador puede generar un reporte de cierre.
15. Ollama puede redactar el resumen final del reporte.

---

Pregunta frecuente: ¿Cuál es el flujo completo de un trámite automático?

El usuario solicita un servicio, el sistema crea el pedido, guarda input_data, valida y descuenta saldo, revisa si el servicio es automatic, despacha ProcessExternalOrderJob, procesa con ExternalOrderAutomationService, usa DhruApiService si el proveedor es dhru, genera o recibe el resultado, actualiza api_status, guarda api_report, marca processed_by_api y permite descargar el archivo desde el pedido.
